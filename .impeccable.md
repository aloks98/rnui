## Design Context

### Users
- **Primary user**: Alok (sole developer) — building personal projects across dashboards, SaaS products, admin panels, and consumer-facing apps
- **Context**: Internal component library for rapid development across diverse project types
- **Job to be done**: Ship polished, distinctive UIs fast without rebuilding primitives each time

### Brand Personality
**Modern, different, quirky**

- Should feel fresh and opinionated, not generic
- Personality comes through in subtle details — micro-interactions, unexpected color choices, clever component compositions
- Technical sophistication without being sterile

### Aesthetic Direction
- **Visual tone**: Minimal foundation with expressive color — clean structure, bold accents
- **References**: Best-of-all-worlds approach — Linear's polish, Vercel's clarity, Stripe's depth, Notion's composability. Take the strongest quality from each, don't clone any one
- **Anti-reference**: Generic shadcn/ui sites. The default shadcn look (gray/zinc, identical layouts, same card patterns) is explicitly what this should NOT look like. Every project built with rnui should feel distinct
- **Theme**: Dark and light mode both supported. Multiple built-in color themes with excellent customization support
- **Color philosophy**: Move beyond monochromatic — embrace rich, vibrant palettes using OKLch. Ship many built-in themes so projects can have unique identities out of the box

### Design Principles

1. **Never look like default shadcn** — Same primitives, completely different personality. If a component could be mistaken for stock shadcn/ui, it needs more character.

2. **Color is a feature, not an afterthought** — Rich built-in themes, vibrant semantic colors, OKLch for perceptual consistency. Users should have many palette options that all feel intentional.

3. **Quirky in the details** — Personality lives in micro-interactions (the `active:translate-y-px` press effect), unexpected but functional touches, and opinionated defaults. Not random — deliberately different.

4. **Minimal structure, maximal expression** — Clean component APIs and lean markup. The complexity budget goes into visual richness and interaction quality, not configuration overhead.

5. **Accessibility when possible** — No strict WCAG target, but accessible patterns are already baked in via Base UI primitives. Maintain what's there, improve when touching components. Good accessibility is good design.
